# Issue Fix Development Skill

这个skill用于处理本项目的issue修复和功能开发。

## 触发条件

当用户明确要求你：
- 修复某个issue
- 基于某个issue进行开发
- 完成某个issue号对应的任务

## 工作流程

⚠️ **流程顺序严格执行** - 以下步骤必须按顺序执行，**绝不能跳过任何步骤**或提前执行后续步骤。

### 1. 确认issue信息
- 确保用户提供本项目的issue链接或issue编号
- 如果没有提供，向用户索取

### 2. 读取issue内容
- 使用GitHub MCP工具读取issue的详细信息（如`mcp__plugin_github_github__issue_read`）
- 从标题、描述和讨论区中确认开发内容
- 对于开发范围、实现效果不明确的部分，**必须**向用户提问

### 3. 整理开发范围
- 将所有明确的内容整理成清晰的开发范围描述
- 向用户进行最终确认
- 只有在用户明确确认后，才能进入步骤4

### 4. 制定开发计划
- **STOP: 在执行此步骤前，必须先完成步骤1-3**
- 切回main分支，并拉取最新代码
- 基于确认的开发范围，制定详细的开发计划
- 计划包含：
  - 功能描述
  - 预期效果
  - 大致的修改点
- **关键步骤：必须将计划以markdown格式评论到issue的评论区中**
  - 这是**重要的设计归档手段**，将设计决策永久记录在issue中
  - 即使后续用户修改或调整计划，原计划也作为历史记录保留
- 计划结尾添加：
  > 本开发计划由Claude Code生成。生成时间{timestamp}。使用模型{model_name}。

### 5. 确认开发计划
- 询问用户计划是否OK
- 如果不OK，根据反馈修改计划并刷新issue评论
- 重复此过程直到用户满意
- 只有在用户明确确认后，才能进入步骤6

### 6. 创建开发分支
- **STOP: 必须先完成步骤5，获得用户明确确认后才能执行此步骤**
- 从main分支拉出新的开发分支，并切换到开发分支
- 分支命名建议：`fix/issue-{number}` 或 `feature/issue-{number}`

### 7. 创建开发checklist
- 基于开发计划创建详细的checklist
- 使用TaskCreate工具管理任务

### 8. 依次开发调试
- 按照checklist顺序执行
- 每个任务开发完成后执行`npm test`
- **每次commit前必须保证npm test通过**
- 建议新增对应的测试用例（不强求）
- 期间不切换分支

### 9. 最终测试
- 所有内容开发完成后，再次执行`npm test`
- 确保所有用例通过
- 如果用例失败，定位问题并修复

### 10. 提交PR
- 将开发分支代码推送到远端对应分支
- 创建PR，Assignees设为 tongl2
- PR内容包含：
  - 关联对应的issue
  - 修改点说明
  - 影响性分析
  - npm test测试报告

## 重要注意事项

- 每次commit前必须通过测试
- 遇到不明确的需求必须向用户确认，不要自行推断
- **永远不能**使用force push/commit amend等高危命令
- ⚠️ **最关键的规则**：
  - 必须严格按照步骤顺序执行
  - **步骤4中，必须将开发计划评论到issue评论区** - 这是重要的设计归档手段
  - 步骤5（确认开发计划）**必须**获得用户明确确认后才能进入步骤
  - 绝不能在发布计划前就创建分支、修改代码或执行任何开发操作
  - 如果用户没有明确说"可以开始"、"计划OK"、"开始开发"等类似确认，不要开始开发
